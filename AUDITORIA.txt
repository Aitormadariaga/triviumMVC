 CRITICO   (bloqueantes antes de produccion)
=====================================================================

[ ] #1  Contrasena admin hardcodeada
        Archivo: PacienteDBHelper.java:273
        Codigo : hashPassword("admin123")
        Impacto: cualquiera que descompile el APK accede como admin.
        Fix    : generar hash aleatorio en primer arranque y guardar
                 en un fichero privado de la app.

[ ] #2  BD medica sin cifrar en almacenamiento publico
        Archivo: PacienteDBHelper.java:161-173 + LoginActivity.java:49-82
        Ruta   : Environment.getExternalStoragePublicDirectory(
                    DIRECTORY_DOCUMENTS)/TriviumData/trivium.db
        Impacto: incumple GDPR y scoped storage de Android 10+.
                 Cualquier app con acceso a almacenamiento puede leer
                 pacientes, sesiones y usuarios.
        Fix    : migrar a getFilesDir() + SQLCipher.

[X] #3  Cascadas de borrado sin transaccion (scope ampliado)
        Archivo: PacienteDataManager.java - eliminarPaciente + eliminarSesion
        Aplicado: beginTransaction / setTransactionSuccessful / endTransaction
                  en ambos metodos. Ademas se borra usuario_sesion antes de
                  sesiones (evita huerfanos en la tabla intermedia) y se hace
                  rollback si el paciente/sesion no existia. Simplificado
                  SesionController.eliminarSesion (delegaba en dos llamadas
                  separadas, ahora confia en la transaccion del DataManager).
        Commit : decccbe

[X] #4  NPE en callback BT tras desconexion inesperada
        Archivo: MainActivity.java:393
        Codigo : dispositivo1.getBtDevice().getName()
        Impacto: app se cierra al pulsar Conectar si el dispositivo se
                 desconecta justo tras el callback de conexion.
        Commit : 55a2eeb  (2026-04-22)

[ ] #5  Fallback a contrasena en texto plano si MessageDigest falla
        Archivo: PacienteDBHelper.java:293
        Impacto: si hashPassword() falla por cualquier motivo, guarda
                 la contrasena SIN HASH en la BD.
        Fix    : lanzar excepcion en vez del fallback.

[ ] #6  allowBackup="true" + MANAGE_EXTERNAL_STORAGE en manifest
        Archivo: AndroidManifest.xml:31, 36
        Impacto: backup via ADB expone todos los datos clinicos.
                 MANAGE_EXTERNAL_STORAGE es permiso peligroso que ya
                 no hace falta si se migra a almacenamiento privado.
        Fix    : allowBackup=false + quitar requestLegacyExternalStorage
                 + quitar MANAGE_EXTERNAL_STORAGE tras migrar a
                 getFilesDir().

 ALTO   (bugs reales que afectan el uso)
=====================================================================

[ ] #7  Handlers sin Looper.getMainLooper() explicito
        Archivo: TratamientoController.java:43-48
        Codigo : new Handler()  (sin especificar Looper)
        Impacto: solo funciona "por suerte" si se instancia desde UI
                 thread. Frágil ante refactors.
        Fix    : new Handler(Looper.getMainLooper()).

[X] #8  IndexOutOfBounds en lectura de bateria
        Archivo: TratamientoController.java:431
        Codigo : while (recBuffer[j] < 2 || recBuffer[j] > 3) j++
        Impacto: crash de toda la app si llegan bytes BT sin header
                 valido (ruido, respuestas de otros comandos, buffer
                 residual tras reconexion).
        Commit : 55a2eeb  (2026-04-22)

[ ] #9  Busy-wait bloqueante en UI thread
        Archivo: TratamientoController.java:234
        Codigo : for (int s = 60000; s > 0; s--) ;
        Impacto: consume CPU al 100% durante el delay y congela la UI.
        Fix    : reemplazar por Thread.sleep(delay) en thread aparte,
                 o un postDelayed.

[ ] #10 Memory leak de Runnable de timer
        Archivo: TratamientoController.java:242-250, 385-481
        Impacto: el closure del Runnable captura "dispositivo"; si la
                 Activity se destruye sin llamar a detenerTimer(), el
                 handler sigue referenciando objetos muertos.
        Fix    : detenerTimer() en onDestroy / ViewModel con scope
                 lifecycle-aware.

[ ] #11 Lambdas de callback BT capturan "this" implicito
        Archivo: MainActivity.java:387-412
        Impacto: si la conexion tarda y la Activity muere antes del
                 callback, hay referencia circular -> leak.
        Fix    : WeakReference a Activity, o comprobar isDestroyed()
                 al entrar al callback.

[ ] #12 Resource leak en leerArchivoMACs()
        Archivo: BluetoothController.java:86-106
        Impacto: InputStream / BufferedReader se dejan abiertos si
                 hay excepcion antes del close.
        Fix    : try-with-resources.

[ ] #13 Parse de EditText sin validar rango
        Archivo: MainActivity.java:460-461, 483-484, 687-689
        Codigo : Integer.parseInt(Param3.getText().toString())
        Impacto: acepta valores fuera del limite del hardware (>19)
                 antes de la validacion posterior.
        Fix    : validar rango tras el parse y rechazar con mensaje
                 claro al usuario.

[ ] #14 Enum.valueOf(Genero) sin try-catch
        Archivo: PacienteController.java:646 aprox.
        Impacto: si la BD tiene un valor de genero corrupto / antiguo,
                 crash al leer paciente.
        Fix    : wrapper con default fallback.

[ ] #15 Logs con PII (DNI, nombres) en Log.d()
        Archivo: multiples (~135 logs dispersos)
        Impacto: informacion sensible de pacientes visible en logcat
                 incluso en builds de release.
        Fix    : revisar y eliminar logs con datos de paciente;
                 bajar a Log.v() y compilar fuera en release.

=====================================================================
 OBSERVACIONES ARQUITECTONICAS
=====================================================================

 A. MVC se rompe en MainActivity
    Mezcla UI + gestion de estado de conexion + flags (recentlyConnected1).
    Migrar a ViewModel + LiveData/Flow (AndroidX) resuelve ciclo de vida,
    leaks y parte de la duplicacion a la vez.

 B. Duplicacion dispositivo 1/2 es la mayor deuda (ver #16)
    Imposible soportar un 3er dispositivo sin reescribir.
    Extraer a DispositivoController + coleccion.

 C. Operaciones multi-tabla no atomicas
    Crear paciente + vincular usuario, etc. Envolver en beginTransaction.

 D. Gestion de lifecycle incompleta (ver #17)
    onPause / onStop no paran timers ni desconectan BT.

 E. Almacenamiento externo + requestLegacyExternalStorage (ver #2, #6)
    Combina problema legal (GDPR), tecnico (scoped storage) y de
    mantenimiento. Es el punto mas urgente del bloque critico.
